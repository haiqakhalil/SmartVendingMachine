Java OOP Spring Boot REST Next.js Frontend MySQL Database MVC Architecture 4-Week Plan
Core features
🛒
Product Catalog
Browse items with name, price, category, and real-time stock count. Low-stock badge when quantity drops below 2.
💳
Multi-Payment
Cash (with change calculation), Card (card number input), and NFC tap-to-pay — all three methods supported via polymorphism.
📦
Inventory Management
Admin can add, edit, delete, and restock items. Every restock event is logged with timestamp and operator name.
📊
Sales Dashboard
Full transaction history with item name, amount, payment method, and date. Pulled live from the MySQL transactions table.
🔐
Operator Login
Secure admin login with hashed passwords. Only authenticated operators can access the admin dashboard and restock panel.
📝
Transaction Log
Every purchase is saved to MySQL and also appended to sales_log.txt as a human-readable backup with timestamp.
Architecture overview
🎨 Frontend: Next.js (React + TypeScript) — browser-based vending machine UI with item cards and payment modal
⚙️ Backend: Java Spring Boot — REST API with 5 controllers exposing /api/items, /api/vending/buy, /api/transactions, /api/operators, /api/restock
🗄️ Database: MySQL via Spring Data JPA — 4 tables: items, transactions, operators, restock_log
📁 File handling: FileHandler.java using BufferedWriter appends every transaction to sales_log.txt
🔗 Frontend-Backend bridge: REST API over JSON — Next.js calls Spring Boot endpoints using fetch() in api.ts

OOP concepts demonstrated — each one mapped to your project
Encapsulation
All fields in Item.java, Operator.java, Transaction.java are private. Public getters/setters control access.
Abstraction
PaymentMethod is an interface — it defines what payment must do without exposing how each type works.
Polymorphism
VendingController calls payment.pay() — works identically for Cash, Card, and NFC without if-else chains.
Inheritance
CashPayment, CardPayment, and NFCPayment all inherit the PaymentMethod contract via implements.
Collections
VendingMachine uses ArrayList<Item> for inventory. Spring Boot returns List<Transaction> from the database.
Exception Handling
try-catch blocks in FileHandler.java (file I/O), DatabaseHelper.java (JDBC), and all REST controllers.
File I/O
FileHandler.java uses BufferedWriter to append transactions to sales_log.txt. Demonstrates Java file handling.
Interfaces
PaymentMethod interface enforces pay() and getPaymentType() on every payment class — design by contract.
MySQL database — 4 tables
🛒 items
id          INT    AUTO_INCREMENT PK
name        VARCHAR(100)
price       DECIMAL(10,2)
quantity    INT
category    VARCHAR(50)
All product data. ItemController reads and writes this table. Stock is updated after every purchase.
💳 transactions
id             INT    AUTO_INCREMENT PK
item_name      VARCHAR(100)
amount         DECIMAL(10,2)
payment_type   VARCHAR(20)
purchased_at   DATETIME
Every purchase is recorded here. Admin dashboard fetches all rows to display sales history.
🔐 operators
id            INT    AUTO_INCREMENT PK
name           VARCHAR(100)
username       VARCHAR(50)
password_hash  VARCHAR(255)
role           VARCHAR(20)
Admin/staff accounts. Passwords are hashed before storage. Role is "ADMIN" or "STAFF".
📦 restock_log
id              INT    AUTO_INCREMENT PK
item_id         INT    FK → items
operator_id     INT    FK → operators
quantity_added  INT
restocked_at    DATETIME
Audit log for every restock. Tracks who added stock, how much, and when.
REST API endpoints
GET /api/items             → fetch all products for the vending UI
POST /api/items             → admin adds a new item
DELETE /api/items/{id}     → admin deletes an item
POST /api/vending/buy      → customer buys an item (payment + stock update)
GET /api/transactions     → admin fetches all sales history
POST /api/operators/login  → admin login with credentials
POST /api/restock          → operator restocks an item
GET /api/restock/logs     → admin views all restock history
